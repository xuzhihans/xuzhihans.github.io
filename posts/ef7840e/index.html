细说Hexo结合Github Pages搭建个人博客

发表于 2019-09-25 更新于 2019-09-26

    1.1 Hexo是什么？

Hexo is a fast, simple & powerful blog framework powered by Node.js

按照上面Hexo官方的说法，Hexo是一个用Node.js语言写的，功能强大并且简便，能够快速安装部署的博客框架。
简单来说，Hexo就是一个让你自己搭建博客的工具，这个工具确实非常流行，很多技术人员都使用这个博客系统搭建博客并进行写作。

1.2 关于博客的“废话”

题外“废话”（不想看的可掠过），目前你要使用博客写作，一般有两种方案可选：

  • 使用已有的博客网站
    这种方式就是别人已经帮你搭建好了博客系统，你只需要登陆这个网站，然后注册个账号，就可以开始写作了。
    比较有名的技术博客站点如：
    CSDN博客，博客园
    还可以使用一些“某笔记”软件进行记录，如：
    有道笔记，印象笔记……

  • 自建博客平台
    如果使用这种方案，基本上你需要做两件事情：

    • 找一套合适的博客框架（就是博客软件）安装，知名度比较高的有以下几种：
      • Wordpress：之前很多人用，近些年不流行了
      • Hexo+Github Pages：目前非常流行，功能满足绝大多数需求
      • Sphinx+Pandoc+GitHub+Readthedocs：Sphinx是Python语言写的，是动态博客系统，感觉部署起来有点复杂
      • TypeEcho：动态博客系统

    有了上面的软件以后，要找一个地方能够把软件安装并运行起来，并且还需要向外提供一个网址，让你和别人通过浏览器进行访问，从而查看博客内容。那么就需要一个“环境”，这种环境基本上分为以下几类：

    • 自建
      这种方法，你需要自己购买服务器（可以是云虚拟机），购买ip地址，购买域名，然后在这个服务器（或者虚拟机）中安装博客系统，用于访问你的博客网站，说白了就是建立自己的网站。
    • 托管
      比如使用GitHub Pages（GitHub静态页面），这种方式优点是免费，你不需要自己购买服务器，不需要自己申请公网ip地址，也不需要自己购买域名，你直接在github上创建一个账号，在该账号下创建一个“项目”，这个“项目”可以有一个网址，然后把你的“博客系统”上传到这个“项目”中去，就可以使用浏览器开始访问了。

个人倒腾过上述所有方法，最终还是回归Hexo+GithubPages方案，毕竟这种方案（辅以各种强大插件）能够满足技术博客绝大多数功能需求，（辅以各种强大的主题theme）也能给观看者带来赏心悦目的阅读快感。
OK，废话基本到此，接下来进入正题。

5. 将本地博客系统发布到网上

通过上面的步骤，我们已经在本地建立好了博客系统，并且也知道了如何利用博客系统写文章，并在本地发布、访问。
接下来的问题是，我们写的博客文章需要随时能查看，并且也有可能需要让更多的人来看到内容，以达到知识分享的目的，所以我们就需要将我们本地的博客系统“搬”到网上。
这里我们将结合Github Pages功能，将我们本地的博客系统“托管”到Github上，从而实现我们随时查看和分享博客内容的目的。

5.1 hexo博客系统与Gihub Pages结合的基本原理

5.1.1 什么是Github Pages？

Github Pages是Github官方提供的一个静态页面托管虚拟主机。说白了就是，它免费提供我们一个空间，我们把自己写好的html代码上传到这个空间中，Github Pages就能够作为服务器，把html内容提供给访问服务器的人（具体说是浏览器）。
所以，Github Pages主要做了这么几件事情：

  • 免费提供给你一些空间，用于存放你上传的代码（html相关的代码）
  • 免费提供给你一个域名，用于你访问html页面（在这里也就是我们的博客网页）
  • 它作为一个网站服务器，根据我们的html代码提供网站服务

5.1.2 怎么把本地的博客系统提交到Github Pages上？

Github是一个代码管理仓库，它使用git工具（命令）来同步本地和Github仓库（网上）的代码，如果本地代码比网上Github仓库中的代码更新（有变动），那么在执行git相关命令时，就会自动的将本地代码（有变动部分增量式更新）更新到Github仓库里。
这里还有一个关键的问题:

之前说了，我们现在也明白，我们写的博客文章其实就是一个以.md结尾的文本文件，该文件内容其实是按照Markdown语法写的，但是Github是一个html服务器，那么文本文件怎么会变成html网页的？

这个很重要，这确实需要将Markdown语法的文本文件转换成html代码，这个转换的动作需要我们执行hexo相关命令后hexo帮我们做的。
大致的过程是：当我们执行hexo generate命令后，hexo会拿着我们写好的.md结尾的文本文件，结合hexo默认提供的（也可以自行修改）css样式等一些文件，自动的帮我们“生成”（渲染）出相关的html文件。
git命令提交的内容中就包括了渲染后的html文件给Github Pages，而后Github Pages就可以利用这个html代码来提供网站服务了。
这里可以先说一下，在我们站点目录（这里是blog目录）下，当我们使用hexo generate命令后，会创建一个public/ 的目录，在该目录下，还会创建一个posts/ 目录，在这个目录下，就存放着渲染过后的html文件。一个.md的文本文件，在渲染过后就对应一个渲染过后的html文件。

OK，在了解了基本原理之后，我们可以来实现渲染及向Github Pages提交代码（deploy, 部署）的功能了。

5.2 将本地博客系统发布到Github Pages上

根据之前的介绍，要想把本地博客发布到Github Pages上，分别需要用到以下工具和进行以下操作：

  • git工具
    git工具一开始我们就已经安装过了，这里并不需要再安装git了。
    是将本地代码（文件）同步至Github的工具，在这里，我们并不需要手动使用git工具进行操作。我们只需要使用hexo deploy命令，而hexo deploy命令会调用git命令去执行同步操作。
    但是，我们必须对git工具进行必要的配置。

  • 在本地服务器生成SSH公钥，而后复制粘贴到Github
    我们说了，git工具是用来同步本地文件与Github仓库所使用的，但是在同步之前，Github仓库必须认为我们本地服务器发起的同步操作是被允许的，是合法的。那么在同步一开始就需要先有一个认证机制。同步的这个协议是由ssh来完成的，认证方法有很多种，这里使用的也是推荐的方法是利用ssh公钥的方式进行认证。
    也就是说，先在本地服务器上产生一对用于SSH的密钥（1个公钥，1个私钥），然后把公钥放到Github网站上。
    那么随后的通信中，本地git要把文件同步到github仓库时，会携带上公钥给github服务器，当github服务器看到这个公钥是之前存储过的后，认为这个操作是合法的，也就通过了认证，接下来就可以同步数据了。

  • 设置hexo站点配置文件(site config file)：blog/_config/yml

  • 安装hexo-deployer-git
    这是hexo的一个“插件”，hexo中有很多“插件”来帮我们单独完成某些工作。
    这个插件从字面上就可以看出，是与git有关的。没错，其实在使用hexo deploy把本地已经生成的“静态网页”内容，调用git工具，上传到Github Pages中去。
    调用逻辑基本是：
    hexo deploy ---调用---> hexo-deployer-git ---调用---> git ---> 将“静态网页”文件上传到Github上。

  • hexo generate命令
    这个命令就是将我们写好的.md文档，结合hexo的css样式等东西，在本地生成“静态网页”相关文件，这些文件后续可以通过hexo deploy命令部署到Github Pages上去。

  • hexo deploy命令
    将生成的本地html网站文件上传到Github仓库
